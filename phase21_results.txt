KEY FINDINGS FROM PROFILING:
  1. Python/numpy overhead per token is ~1ms (NOT 56ms as hypothesized)
  2. The 56ms gap is NOT from Python dispatch — it's from GEMV arithmetic intensity
  3. Native ops ARE already wired in forward_layer() — no wiring fix needed
  4. The remaining gap is purely from the Q4×Q8 GEMV kernel vs llama.cpp's Q4_K_M
  5. llama.cpp's Q4_K_M uses superblock quantization (16-element groups within 128-element blocks)
     which has better compression and fewer scale lookups than our per-group Q4×Q8 format
  6. Our Q4×Q8 format reads more bytes per GEMV call (scales + weights) than llama.cpp's Q4_K_M

ROOT CAUSE OF REMAINING GAP:
  The 0.45 tok/s gap is NOT a framework overhead issue — it's a quantization format
  efficiency issue. llama.cpp's Q4_K_M format achieves better memory bandwidth utilization
  because:
  - Q4_K_M: 128-element superblocks with 16-element sub-blocks → fewer scale reads
  - Our Q4×Q8: 32-element groups with per-group scales → more scale reads
  - At 24 GB/s bandwidth, every extra byte read reduces throughput

  This is a fundamental algorithmic difference, not a code quality issue.
