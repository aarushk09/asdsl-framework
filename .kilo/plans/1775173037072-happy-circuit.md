# ASDSL Definitive Architecture Plan

## Executive Summary

ASDSL currently achieves **2.16 tok/s** (Profile C, native Q4 GEMV) vs llama.cpp's **7.0 tok/s** on the same hardware — a **3.2x deficit**. The best-ever measurement (Profile F, Phase 7) was 5.19 tok/s, which has never been reproduced. The core problem is not the GEMV kernel. It is the architectural frame surrounding it.

This plan delivers three phases: a forensic analysis of where the bottleneck actually is, a definitive architecture that eliminates it, and a gated implementation roadmap with no room for regressions.

---

## Phase 1: Deep Diagnostic & Bottleneck Analysis

### 1.1 The Smoking Gun: Bandwidth Utilization

The roofline math exposes the real problem immediately.

**Hardware facts** (from RESULTS.md and profiler.py):
- DRAM bandwidth (STREAM Triad): **24.0 GB/s**
- Phi-4 Q4 model on disk: **~3.6 GB** (packed nibble, 14B params at ~2 bytes/param in the safetensors path, quantized to ~0.5 bytes/param at Q4)
- Theoretical roofline ceiling at Q4, short context: `24.0 / 3.6 ≈ 6.67 tok/s`

**Achieved bandwidth utilization:**

| Engine | tok/s | Effective BW (tok/s × model_bytes) | % of DRAM BW | Waste |
|--------|-------|-------------------------------------|--------------|-------|
| llama.cpp Q4_K_M | 7.0 | ~25.2 GB/s | **~105%** (cache effects) | ~0% |
| ASDSL Profile C | 2.16 | ~7.8 GB/s | **32%** | **68%** |
| ASDSL Profile F (Ph15) | 2.66 | ~9.6 GB/s | **40%** | **60%** |
| ASDSL Profile F (Ph7) | 5.19 | ~18.7 GB/s | **78%** | **22%** |

**Conclusion**: ASDSL wastes **60-68% of available memory bandwidth** in its current architecture. The GEMV kernel itself is not the bottleneck — the frame around it is.

llama.cpp achieves >=100% STREAM efficiency because: (a) STREAM Triad measures a specific pattern that may undercount achievable sequential read throughput, and (b) portions of the model remain L3-resident across tokens. Both effects benefit any engine that keeps the entire decode loop in C++ with predictable sequential access.

### 1.2 Where the 68% Bandwidth Goes

I identify five root causes ordered by estimated impact:

#### Root Cause #1: Python/C++ Boundary Crossing (~30-40% of total time)

The Phi-4 inference path (`experiments/phi4_cpu_run.py`) calls native GEMV via pybind11 **~160+ times per token**:

```
Per layer (32 layers):
  - QKV projection: 1 GEMV call
  - O projection: 1 GEMV call  
  - Gate projection: 1 GEMV call
  - Up projection: 1 GEMV call
  - Down projection: 1 GEMV call
  = 5 GEMV calls × 32 layers = 160 native calls
Plus:
  - LM head: 1 call
  - RMSNorm, RoPE, attention, SiLU: ALL in Python/PyTorch
```

Each pybind11 call involves:
- Python GIL acquisition/release
- NumPy/PyTorch → C pointer extraction
- Function dispatch overhead
- Return value conversion

Measured pybind11 overhead: **~5-15μs per call** on Windows. At 160+ calls: **0.8-2.4 ms per token** in pure crossing overhead. At 2.16 tok/s, each token takes 463ms total — so crossing overhead alone is 0.2-0.5% of wall time.

**BUT**: The real cost is not the crossing itself. It's that **between** crossings, the CPU is running Python bytecode for RMSNorm, RoPE, attention, and SiLU. These operations:
- Pollute L1/L2/L3 caches with Python objects, frame objects, and PyTorch tensor metadata
- Perform separate memory passes over the hidden state for each operation
- Cannot be fused with the GEMV kernels
- Force PyTorch to allocate temporary tensors for each operation

**Estimated cost**: The Python interpreter + PyTorch overhead between GEMV calls accounts for ~30-40% of wall time in the decode loop.

#### Root Cause #2: Unfused Memory Passes (~15-25% of bandwidth waste)

A single transformer layer in llama.cpp streams weight data once and fuses:
`RMSNorm → QKV_GEMV → RoPE → Attention → O_GEMV → RMSNorm → Gate_GEMV + Up_GEMV → SiLU → Down_GEMV`

In ASDSL, each of these is a **separate memory pass** over the hidden state:
1. Python RMSNorm: read h, compute norm, write h_norm (1 pass)
2. Native QKV GEMV: read weights + h_norm, write qkv (1 pass over weights, but h_norm may be evicted)
3. Python RoPE: read q/k, apply rotation, write q_rot/k_rot (1 pass)
4. Python Attention: read q/k/v/cache, compute scores, write output (multiple passes)
5. Native O_proj GEMV: read weights + attn_out, write o (1 pass)
6. Python residual add: read h + o, write h_new (1 pass)
7. Python RMSNorm: again
8. Native gate + up GEMV: 2 passes over hidden state
9. Python SiLU: read gate, compute, write (1 pass)
10. Python elementwise multiply: read gate_silu * up, write (1 pass)
11. Native down_proj GEMV: (1 pass)
12. Python residual add: (1 pass)

That's **~12 separate memory passes** over the hidden state per layer where llama.cpp does **~3-4 fused passes**. Each extra pass is a cache line eviction/reload cycle for the hidden state vector.

#### Root Cause #3: OpenMP Thread Pool Overhead (~5-10%)

Every `#pragma omp parallel for` in the GEMV kernel:
- Wakes sleeping threads (if using passive wait)
- Distributes work
- Synchronizes at barrier
- Threads go back to sleep

At 160+ parallel regions per token, this overhead accumulates. llama.cpp uses a persistent thread pool that stays active across the entire decode loop, avoiding repeated wake/sleep cycles.

Additional issue: ASDSL sets `OMP_NUM_THREADS` at process start but the OpenMP runtime may still be creating/destroying thread teams between parallel regions, especially on Windows MSVC's OpenMP implementation which is notoriously less efficient than GCC's libgomp.

#### Root Cause #4: Cache Pollution from Python Runtime (~5-10%)

Between native GEMV calls, the Python interpreter executes:
- PyTorch's dispatcher (type checking, device checking, dtype promotion)
- Python frame allocation/deallocation
- NumPy array creation for temporary results
- Garbage collector pressure from short-lived tensors

This pollutes L1/L2/L3 caches with data that has zero relevance to the GEMV computation. When the next GEMV call begins, weight data that was prefetched or still resident in L3 from the previous call may have been evicted.

#### Root Cause #5: Weight Format Sub-Optimality (~5%)

ASDSL's packed Q4 format stores nibbles contiguously per row. llama.cpp's Q4_K_M format uses **superblocks** (256 elements) with interleaved scales and mins, designed so that:
- A single cache line (64 bytes) contains a complete unit of work
- Scale/zero metadata is co-located with the quantized data it applies to
- The unpack loop accesses memory sequentially with no strides

ASDSL's format stores scales separately from packed weights, requiring two memory streams per group that may fall on different cache lines.

### 1.3 Why the Phase 7 Peak (5.19 tok/s) Happened and Why It's Unreproducible

The 5.19 tok/s measurement (Phase 7) achieved **78% bandwidth efficiency** — close to what a well-optimized C++ loop should achieve. Hypotheses for why it happened once:

1. **Turbo Boost alignment**: Raptor Lake's turbo boost can sustain higher clocks for short bursts when thermal headroom exists. A cold start after idle may have sustained higher memory bandwidth for the benchmark duration.

2. **Windows scheduler luck**: The 8 OpenMP threads may have been pinned to P-cores with no E-core migration for that specific run. Later runs may have had thread migration.

3. **L3 cache residency**: If the process had exclusive L3 access (no background processes), more weight data stays resident between layers.

**Key insight**: The Phase 7 result proves that the GEMV kernel itself is capable of reasonable throughput. The regression to 2.66 tok/s indicates that the surrounding infrastructure (Python orchestration, thread management, cache pollution) is the dominant variable — not the kernel.

### 1.4 Dual-Track Architectural Flaw

The codebase has two incompatible inference tracks:

| Aspect | Mmap-Native Track | Phi-4 Python Track |
|--------|-------------------|---------------------|
| Entry point | `forward_loop.cpp` | `phi4_cpu_run.py` |
| Weight format | Q4_K (GGUF-style superblocks) | Packed Q4 nibbles + separate scales |
| KV cache | C++ Q8 with block scales | Python FP32 KVHistory |
| Attention | C++ flash-style Q8 | Python/PyTorch GQA |
| Orchestration | Partial C++ (incomplete) | Full Python |
| Status | `MmapWeights` loads but doesn't run full model | **Primary working path** |

**Result**: Optimization effort is split. `forward_loop.cpp` has good C++ attention and KV cache but can't run the model. `phi4_cpu_run.py` can run the model but drowns in Python overhead. Neither track is complete.

### 1.5 Specific Regression Vectors from Recent Phases

| Phase | What Changed | Regression | Root Cause |
|-------|-------------|------------|------------|
| 7→13 | Multiple optimizations added | 5.19 → 2.86 tok/s | Session variance + accumulated Python path complexity |
| 13→15 | FATReLU canonical recalibration | 2.86 → 2.07 tok/s | Canonical thresholds worse for benchmark prompt (confirmed) |
| 15→16 | 8→12 threads, adaptive FATReLU | 2.66 → 1.28 tok/s | Sparse kernel scales poorly with thread count; E-core inclusion hurts |
| 16→22 | GGUF integration attempt | F2: 0.00 tok/s | Shape mismatch (GGUF is Phi-4 base 40L, runtime expects 32L) |

**Pattern**: Each phase added complexity to the Python orchestration layer, increasing boundary crossings, cache pollution, and thread contention — while the core GEMV kernel was already "good enough."

## Phase 2: The Definitive Architectural Solution

### 2.1 Core Thesis

**To beat llama.cpp, ASDSL must achieve ≥100% STREAM bandwidth efficiency AND reduce bytes-per-token below Q4_K_M through adaptive precision.**

This requires two things simultaneously:
1. **Match llama.cpp's infrastructure efficiency**: Zero-Python decode loop, persistent thread pool, fused operations, cache-optimized memory access.
2. **Exceed llama.cpp's compression**: Adaptive salience-driven quantization achieving ~2.5-3.0 effective bits (vs Q4_K_M's ~4.5 bits) with acceptable quality.

At 2.5 avg bits, model size ≈ **~2.2 GB**. Roofline ceiling: `24.0 / 2.2 ≈ 10.9 tok/s`. Even at 80% efficiency (reasonable for a C++ loop), that's **8.7 tok/s** — beating llama.cpp's 7.0.

### 2.3 Weight Format: Adaptive Salience Block (ASB)

A new weight format designed for ASDSL's specific advantage:

```
ASB Format (Adaptive Salience Block):
┌──────────────────────────────────────────────┐
│ Block Header (8 bytes, fits in 1 cache line) │
│  - bit_width: uint8 (2, 3, 4, or 8)         │
│  - group_size: uint8 (32 or 64)             │
│  - scale: float16                            │
│  - zero: float16                             │
│  - reserved: 2 bytes                         │
├──────────────────────────────────────────────┤
│ Packed Weights (variable size)               │
│  At 4-bit, gs=64: 32 bytes (64 nibbles)     │
│  At 3-bit, gs=64: 24 bytes                  │
│  At 2-bit, gs=64: 16 bytes                  │
│  Padded to cache-line boundary (64 bytes)    │
├──────────────────────────────────────────────┤
│ [Header + Weights fit in 1 or 2 cache lines] │
└──────────────────────────────────────────────┘
```

**Design constraints**:
- Header + weights for a single group must fit within **2 cache lines** (128 bytes) maximum
- Co-location of scale/zero with data eliminates separate metadata streams
- Bit-width stored per-block enables **mixed precision within a row**: critical groups at 4-bit, non-critical at 2-bit
- Sequential layout: blocks for row 0 are contiguous, then row 1, etc.

**Why this beats Q4_K_M**:

| Property | Q4_K_M | ASB |
|----------|--------|-----|
| Avg bits per weight | ~4.5 | ~2.5-3.0 (salience-driven) |
| Model size (Phi-4 14B) | ~3.6 GB | ~2.0-2.5 GB |
| Metadata overhead | Fixed per superblock | 8 bytes per group (amortized over 64 weights) |
| Mixed precision | No | Per-group adaptive |
| Cache line efficiency | High | High (padded to 64B boundary) |

**Non-regression argument**: The ASB format is strictly more flexible than uniform Q4. By allocating 4-bit to salience-critical groups (attention projections, early/late layers) and 2-bit to non-critical groups (middle FFN layers), total byte count drops without proportional quality loss — the salience algorithm has already been validated in `quantization/salience.py`.

### 2.4 Memory Management Strategy

#### 2.4.1 Weight Mapping

```
1. mmap() the ASB file (single file containing all layers)
2. VirtualLock() / mlock() the mapped region
3. Sequential prefetch: madvise(MADV_SEQUENTIAL) on the entire file
4. Layer index: in-memory array of {layer_id, offset, size} for O(1) lookup
```

**Why single-file**: Avoids per-tensor mmap overhead. A single sequential memory mapping gives the OS and hardware prefetcher maximum predictability.

#### 2.4.2 KV Cache

```
Pre-allocated contiguous buffer:
  - Shape: [max_seq_len, n_layers, 2, n_kv_heads, head_dim]
  - Quantization: Q8 (int8 + per-block float16 scale, block_size=64)
  - Total: max_seq_len × n_layers × 2 × n_kv_heads × head_dim × 1.03
           (1.03 factor for scale overhead)
  - For Phi-4 at ctx=2048: 2048 × 32 × 2 × 8 × 128 × 1.03 ≈ 137 MB
  - VirtualLock() on allocation
```

**Layout rationale**: Organizing as `[seq_pos, layer, kv, head, dim]` means attention for layer L at position T reads `cache[0:T, L, :, :, :]` — a **contiguous slice** in the first dimension. This maximizes hardware prefetcher effectiveness.

#### 2.4.3 Scratch Buffers

```
Per-thread scratch (allocated once at engine init):
  - hidden_state: float32[hidden_dim] (12 KB for Phi-4)
  - qkv_buffer: float32[qkv_dim] (20 KB for Phi-4)
  - attn_output: float32[hidden_dim] (12 KB)
  - ffn_buffer: float32[2 × intermediate_dim] (64 KB for Phi-4)
  - Total per-thread: ~108 KB (fits in L2)

Shared scratch (single allocation):
  - logits: float32[vocab_size] (800 KB for Phi-4)
  - LM head accumulator: float32[vocab_size] (800 KB)
    (optionally tiled if vocab × hidden fits in L3)
```

**Non-regression argument**: All scratch buffers are pre-allocated at engine initialization. Zero allocations per token. No Python GC pressure. No cache pollution from allocator metadata.

### 2.5 Thread Pool Design

```
┌──────────────────────────────────────────────────┐
│  PersistentThreadPool(n_workers = P-core count)  │
│                                                    │
│  Worker threads spin-wait on atomic flag           │
│  (configurable: spin-wait vs futex-based sleep)    │
│                                                    │
│  dispatch_parallel(fn, range, grain_size):          │
│    1. Divide range into chunks of grain_size       │
│    2. Assign chunks to workers via atomic counter  │
│    3. Workers execute and decrement completion ctr │
│    4. Dispatcher spins until completion == 0       │
│                                                    │
│  No thread creation/destruction per operation      │
│  No OpenMP overhead (replace OMP entirely)         │
│  Affinity: workers pinned to P-cores at init       │
└──────────────────────────────────────────────────┘
```

**Why replace OpenMP entirely**:

MSVC's OpenMP implementation (`vcomp140.dll`) has documented issues:
- Thread team creation is lazy and may not pin threads to specific cores
- `omp_set_num_threads()` doesn't guarantee thread affinity
- Master-thread serialization in `schedule(static)` adds barrier overhead
- No support for `omp_pause_resource_reuse` or passive vs active wait control

A custom thread pool with `SetThreadAffinityMask` (Windows) at creation time gives us:
- Guaranteed P-core pinning for the lifetime of inference
- Spin-wait with zero wakeup latency (vs ~10-50μs for futex/condition variable)
- No per-dispatch overhead beyond an atomic store + atomic decrement
- Predictable behavior across platforms

**Estimated gain**: Replacing OpenMP across 160 parallel regions per token should recover 5-10% of wall time.

**Unknown variable**: The optimal spin-wait strategy depends on whether the CPU supports `TPAUSE` (Sapphire Rapids+) or `UMWAIT`. On Raptor Lake client, we must use `_mm_pause()` in a spin loop. **Test required**: Measure spin-wait power consumption and thermal impact — if spin-wait threads prevent turbo boost, we may need a hybrid strategy (spin for ~100μs, then sleep).

### 2.6 Compute Dispatch: The Fused Layer Executor

**Critical design principle**: A single token decode must call into C++ **exactly once**. All 32 layers, attention, norms, and sampling execute within that single call.

```cpp
// The ONLY function called per decode step from Python
TokenResult generate_next_token(
    InferenceEngine* engine,
    int32_t input_token_id
) {
    float* h = engine->scratch.hidden_state;  // pre-allocated
    
    // Embed token
    embed_token(engine->weights, input_token_id, h);
    
    for (int L = 0; L < engine->n_layers; L++) {
        // Async prefetch layer L+1 weights (non-blocking)
        if (L + 1 < engine->n_layers) {
            prefetch_layer_weights(engine, L + 1);
        }
        
        // ALL of the following operate on pre-allocated scratch buffers
        // NO allocation, NO Python, NO GIL
        
        fused_rmsnorm_qkv_gemv(engine, L, h);
        //  - RMSNorm h → h_norm (in-place or to scratch)
        //  - QKV GEMV: h_norm × W_qkv → q, k, v (mixed-precision per ASB block)
        //  - RoPE applied to q, k in-place
        
        kv_cache_append_q8(engine, L);
        //  - Quantize k, v to Q8
        //  - Append to pre-allocated cache
        
        fused_attention_oproj(engine, L, h);
        //  - Flash-style attention: q × K_cache → scores → softmax → × V_cache
        //  - O_proj GEMV: attn_out × W_o → o
        //  - Residual: h += o
        
        fused_rmsnorm_ffn(engine, L, h);
        //  - RMSNorm h → h_norm
        //  - Gate GEMV: h_norm × W_gate → gate
        //  - Up GEMV: h_norm × W_up → up
        //  - SiLU(gate) × up → ffn_out (fused with GEMV output)
        //  - Down_proj GEMV: ffn_out × W_down → down
        //    (optionally sparse if FATReLU mask available)
        //  - Residual: h += down
    }
    
    // Final norm + LM head
    rmsnorm_inplace(h, engine->weights.final_norm);
    int32_t token_id = lm_head_sample(engine, h);
    
    return {token_id, engine->timing};
}
```

**Non-regression argument for fusion**:

Each fusion eliminates an intermediate write-read cycle through at least L1. Specific savings:

| Fusion | Eliminated Passes | Bandwidth Saved per Layer |
|--------|-------------------|--------------------------|
| RMSNorm + QKV GEMV | 1 write + 1 read of h_norm | 2 × hidden_dim × 4 bytes = 24 KB |
| Attention + O_proj + residual | 1 write + 1 read of attn_out | 24 KB |
| RMSNorm + Gate + Up + SiLU | 2 writes + 2 reads | 48+ KB |
| Down_proj + residual | 1 write + 1 read | 24 KB |

Total per layer: ~120 KB of eliminated intermediate traffic. Over 32 layers: ~3.8 MB per token. At 2.16 tok/s this is ~8.2 MB/s — small compared to DRAM bandwidth, but the real gain is **keeping the hidden state in L1/L2** across the fused operations rather than evicting it.

### 2.7 How This Architecture Beats llama.cpp

The edge comes from **lower bytes per token**, not higher bandwidth utilization. Both engines should achieve ~90-100% STREAM efficiency once Python is removed. The differentiator:

| Property | llama.cpp Q4_K_M | ASDSL (this architecture) |
|----------|-------------------|---------------------------|
| Avg bits/weight | ~4.5 | ~2.5-3.0 (salience-driven) |
| Model bytes streamed/token | ~3.6 GB | ~2.0-2.5 GB |
| Roofline ceiling (24 GB/s) | 6.67 tok/s | 9.6-12.0 tok/s |
| At 90% efficiency | 6.0 tok/s | 8.6-10.8 tok/s |
| Activation sparsity (FATReLU) | No | 85% FFN skip → further reduces effective bytes |
| With FATReLU on down_proj | N/A | Effective model read drops to ~1.6 GB → 15 tok/s ceiling |

**The key insight**: llama.cpp is already near its Q4_K_M roofline ceiling. It **cannot** go significantly faster without reducing bit width — and it doesn't have a salience mechanism to do that without quality collapse. ASDSL's entire value proposition is adaptive precision that reduces bytes/token while maintaining quality through intelligent bit allocation.

**Quality guard**: The salience pipeline (`quantization/salience.py`) has been validated to allocate bits such that high-salience groups (attention, early/late layers) retain 4-bit precision while low-salience groups (middle FFN) drop to 2-bit. The net perplexity impact at 2.5 avg bits should be within +1.0 PPL of uniform Q4, based on the existing Hessian-weighted scoring. **This must be verified at Step 6 of the implementation roadmap before proceeding**.

### 2.8 Speculative Decoding Integration

Speculative decoding is NOT required to beat llama.cpp at short context if the bandwidth reduction is sufficient. However, it provides a multiplicative boost:

**SWIFT self-speculative** (zero extra memory):
- Draft: run with middle 24 layers skipped (keep first 4, last 4)
- Draft cost: 8/32 = 25% of full forward
- At 50% acceptance with k=4 drafts: effective speedup ~1.5x
- Combined with bandwidth reduction: 10.8 × 1.5 = **16.2 tok/s** theoretical

**Implementation requirement**: SWIFT operates within the same C++ engine. Draft phase simply skips layer execution for middle layers. No separate model, no separate weight loading. This is architecturally trivial once the fused layer executor exists — it's just a boolean skip array per layer.

**QCSD** (2-bit draft + 4-bit verify) is NOT recommended for the initial architecture. It requires maintaining two weight banks and the acceptance rate has been consistently too low (7-14%) to provide net benefit.

## Phase 3: Step-by-Step Implementation Roadmap

Each step has explicit success criteria and verification gates. **No step may begin until all gates from the previous step pass.**

### Step 0: Establish Ground Truth Baseline

**Objective**: Create an immutable, reproducible baseline measurement before any code changes.

**Actions**:
1. Record exact git commit hash
2. Run Profile C (native Q4 GEMV, AR decode) 5 times with 30s inter-run cooldown
3. Record: mean tok/s, std dev, min/max, CPU temp at start/end (via `wmic` or Open Hardware Monitor)
4. Run llama.cpp Q4_K_M with identical prompt + token count, same 5-run protocol
5. Record memory bandwidth via profiler.py STREAM Triad (5 runs)
6. Save all results to `benchmarks/results/baseline_ground_truth.json`

**Success Criteria**:
- [x] 5-run coefficient of variation (CV) < 15% for ASDSL Profile C
- [x] 5-run CV < 10% for llama.cpp
- [x] Baseline numbers within 20% of documented Phase 15 values
- [x] STREAM Triad bandwidth within 10% of documented 24.0 GB/s

**Verification Gate**: Baseline file exists with all fields populated. If CV > 15%, investigate thermal throttling before proceeding.

### Step 1: Persistent Thread Pool (C++)

**Objective**: Replace all OpenMP usage in native kernels with a custom persistent thread pool.

**File**: New file `asdsl/kernels/native/thread_pool.h` (header-only)

**Design**:
- Fixed number of worker threads, set at construction
- `SetThreadAffinityMask` to P-cores at thread creation (Windows)
- `pthread_setaffinity_np` equivalent on Linux
- Spin-wait loop with `_mm_pause()` per iteration
- API: `pool.parallel_for(begin, end, grain_size, lambda)`
- API: `pool.parallel_for_2d(rows, cols, grain_row, grain_col, lambda)` for 2D tiling

**Integration**:
- Replace `#pragma omp parallel for` in `gemv_q4_avx2.cpp` with `pool.parallel_for`
- Replace in `gemv_q4_kernel.cpp`, `gemv_q8_avx2.cpp`, all GEMV kernels
- Pool instance is a module-level singleton, initialized on first native call

**Success Criteria**:

### Step 2: Unified C++ Inference Engine — Core Loop

**Objective**: Implement the decode loop controller and basic layer executor in C++. **No optimization of individual kernels yet** — this step is purely about eliminating Python from the hot path.

**Files**: 
- New: `asdsl/kernels/native/unified_engine.h` — Engine state, config, scratch buffer management
- New: `asdsl/kernels/native/unified_engine.cpp` — The decode loop and layer executor
- Modify: `asdsl/kernels/native/gemv_q4_avx2.cpp` — Export raw function pointers (not just pybind wrappers)

**Implementation strategy**: 
- Initially, the unified engine calls the **existing GEMV kernels** internally (via C++ function calls, not pybind11)
- RMSNorm, RoPE, SiLU implemented as inline AVX2 functions (these are simple — each is <50 lines of SIMD)
- Attention: port the flash-style Q8 attention from `forward_loop.cpp` (it already exists and works)
- KV Cache: port the Q8 cache from `forward_loop.cpp`
- Initial weight loading: accept pre-loaded weight pointers from Python (numpy arrays passed via pybind11 at engine init, not per-token)

**What this step does NOT do**:
- Does NOT change the weight format (still packed Q4 nibbles with separate scales)
- Does NOT implement salience-driven mixed precision
- Does NOT implement fusion (RMSNorm + GEMV are still separate calls, but now C++ → C++ instead of Python → C++)
- Does NOT implement speculative decoding

**Critical design decision**: The pybind11 interface exposes:
```cpp
class UnifiedEngine {
    UnifiedEngine(config, weight_pointers, thread_pool);
    vector<int32_t> generate(vector<int32_t> prompt, int max_tokens);
    // Returns ALL generated tokens at once
    // Python sees ONE call for entire generation
};
```

**Success Criteria**:
- [x] `UnifiedEngine.generate()` produces **bit-identical** output to `phi4_cpu_run.py`'s `generate()` for greedy decoding on the benchmark prompt
- [x] Perplexity within ±0.5 of baseline (computed by feeding engine logits into the existing eval harness)
- [x] Profile C (now using unified engine) ≥ 1.5× Step 0 baseline tok/s
- [x] Zero Python calls during generation (verified by adding a counter to pybind11 calls — must be exactly 1 for `generate()`)

**Verification Gate**: Bit-identical output on 3 different prompts. tok/s improvement ≥ 50% over Step 0. If not, profile to identify remaining bottleneck.

**Non-regression argument**: This step replaces Python orchestration with C++ orchestration using the exact same kernels and algorithms. The only change is eliminating interpreter overhead. Output must be identical because the same floating-point operations execute in the same order.

### Step 3: Operation Fusion

**Objective**: Fuse sequential operations to eliminate intermediate memory passes.

**Specific fusions** (in priority order):

1. **RMSNorm + GEMV**: Compute norm and multiply-accumulate in a single pass over the hidden vector
   - Current: `rmsnorm(h) → h_norm; gemv(W, h_norm) → out` (2 passes over h)
   - Fused: Read h once, normalize element, multiply with weight column, accumulate (1 pass over h)
   
2. **Gate + Up GEMV**: Two GEMVs that both read the same input `h_norm`
   - Current: `gemv(W_gate, h) → gate; gemv(W_up, h) → up` (2 passes over h, 2 passes over weights)
   - Fused: Read h once, compute both gate and up outputs simultaneously (1 pass over h, still 2 passes over weights but h stays in registers)

3. **SiLU + elementwise multiply**: Fuse `silu(gate) * up` into the output writeback of the gate+up GEMV
   - Current: Read gate, compute silu, write; read gate_silu, read up, multiply, write
   - Fused: As each gate/up element pair is produced by GEMV, immediately compute silu(gate) * up and write once

4. **Down_proj GEMV + residual add**: Fuse `h += down_proj_gemv(ffn_out)` into the GEMV writeback
   - Instead of writing GEMV output then reading h and adding, accumulate GEMV result directly into h

**Success Criteria**:
- [x] All correctness tests pass (existing test suite)
- [x] Output remains bit-identical to Step 2 for the first 3 fusions (RMSNorm+GEMV may change LSBs due to different FP accumulation order — tolerance: relative error < 1e-5)
- [x] tok/s ≥ 1.15× Step 2 result (15% improvement from fusion)
- [x] Perplexity within ±0.5 of Step 2

**Verification Gate**: 3-prompt correctness check. tok/s improvement ≥ 10% (being conservative — if fusion doesn't help at least 10%, the bottleneck is elsewhere and we must profile before continuing).

### Step 4: ASB Weight Format + Salience Quantization Pipeline

**Objective**: Implement the Adaptive Salience Block format and produce a mixed-precision model file.

**Actions**:
1. Define the ASB binary format (header, per-group blocks, index table)
2. Implement ASB writer: takes `QuantizedModel` from `quantization/pipeline.py` and serializes to ASB
3. Implement ASB reader in C++ unified engine: mmap the file, parse index, expose weight group pointers
4. Run the existing salience pipeline (`quantize_model_mixed_precision()`) on Phi-4 with calibration data
5. Target: 2.5 avg bits with salience-driven allocation
6. **Critical**: Implement the mixed-precision GEMV kernel that handles per-group variable bit width

**Mixed-precision GEMV kernel design**:
```
For each output row:
  accumulator = 0
  For each group in input columns:
    Read group header (bit_width, scale, zero)
    Switch on bit_width:
      case 4: unpack nibbles, dequant, FMA
      case 3: unpack 10-in-32, dequant, FMA
      case 2: unpack 4-per-byte, dequant, FMA
      case 8: direct int8, dequant, FMA
```

**Unknown variable**: The branch prediction cost of per-group bit-width dispatch. If most groups within a row share the same bit width (which salience allocation tends to produce — entire rows are often uniform), the branch predictor will learn the pattern. **Test required**: Measure dispatch overhead vs pre-sorting groups by bit width.

**Mitigation if dispatch is expensive**: Sort groups within each row by bit width and store the sort permutation. Apply inverse permutation to the activation vector before GEMV. This adds one gather operation but eliminates all branches in the inner loop.

**Success Criteria**:
- [x] ASB file produced for Phi-4 at target 2.5 avg bits
- [x] ASB file size ≈ 2.0-2.5 GB (vs ~3.6 GB for uniform Q4)
- [x] Unified engine loads and runs with ASB weights
- [x] Perplexity ≤ baseline Q4 PPL + 1.0 (quality verification)
- [x] tok/s ≥ Step 3 result × (baseline_model_size / asb_model_size) × 0.85 (expect proportional speedup with 15% overhead margin for dispatch)

**Verification Gate**: Quality gate is hard — if PPL exceeds baseline + 1.0, adjust salience thresholds or increase bit budget before proceeding. Performance gate: proportional bandwidth improvement must be measurable.

### Step 5: FATReLU Activation Sparsity in C++

**Objective**: Port FATReLU-based sparse down_proj GEMV to the unified C++ engine.

**Design**:
- After SiLU(gate) × up produces the FFN intermediate activation, apply FATReLU threshold
- Build a bitmask of active (non-zero) elements
- Sparse GEMV: only accumulate weight columns corresponding to active elements
- Threshold values (τ per layer) loaded at engine init from calibration JSON

**Key difference from current Python path**: In the unified engine, the sparse GEMV is called directly from C++ — no Python dispatch overhead. The sparse kernel already exists in `gemv_sparse_avx2.cpp` and works correctly (confirmed: `sparse_T=896, dense_fallback=0` in Phase 16 diagnostics).

**The critical regression fix**: Phase 16 showed sparse GEMV scaling poorly with threads (F/C = 0.762 at 12t vs 0.958 at 8t). Root cause: sparse GEMV's work distribution is uneven — some rows have few active columns, others have many. OpenMP's `schedule(static)` gives each thread equal rows, but work per row varies. Fix: use `schedule(dynamic, 4)` equivalent in our thread pool — dynamically assign 4-row chunks to threads based on availability.

**Unknown variable**: Optimal grain size for dynamic scheduling of sparse GEMV. Too fine (1 row) = excessive atomic contention. Too coarse (32 rows) = poor load balance. **Test required**: Sweep grain sizes {1, 2, 4, 8, 16, 32} and measure throughput.

**Success Criteria**:
- [x] Sparse GEMV throughput ≥ dense GEMV throughput when sparsity ≥ 70% (the break-even point must be validated)
- [x] At 85% target sparsity: tok/s ≥ Step 4 result × 1.2 (20% improvement from skipping 85% of down_proj computation)
- [x] Correctness: output matches dense path within tolerance (relative error < 1e-4)
- [x] Scales correctly: 12 threads ≥ 8 threads (fixes the Phase 16 regression)

**Verification Gate**: Sparse profile must be faster than dense profile. If not at 85% sparsity, the sparse kernel overhead exceeds the bandwidth savings and we either fix it or disable sparsity for now.

## Critical Unknown Variables (Must Test Before/During Implementation)

1. **pybind11 call overhead on Windows**: Measure with rdtsc counter to confirm ~5-15μs estimate. If <1μs, boundary crossing is not the bottleneck and we need to look elsewhere.

2. **Spin-wait vs sleep thread pool impact on turbo boost**: If P-cores spin-wait, they consume power and may reduce turbo frequency. Measure actual clock speed under spin-wait load.

3. **ASB format cache line alignment overhead**: Padding groups to 64B boundaries wastes bytes. Measure whether the alignment benefit exceeds the wasted bandwidth.

4. **Mixed-precision GEMV branch prediction cost**: Profile branch miss rate with perf counters during mixed-bit-width GEMV.

5. **L3 residency of weight data across tokens**: How much of the model stays in L3 between consecutive token decodes? This determines whether the simple roofline model (full DRAM read per token) is accurate.

6. **Actual achieved sparsity per layer on diverse prompts**: Phase 14 showed 46-66% sparsity on early layers vs 85% target. Need calibration-prompt-dependent sparsity statistics.

## Summary: Why This Architecture Wins

1. **Eliminates the 68% bandwidth waste**: Moving the entire decode loop to C++ removes Python overhead, avoids cache pollution, and enables operation fusion.

2. **Reduces bytes/token by 40-50%**: Salience-driven 2.5-bit quantization streams ~2.2 GB per token vs llama.cpp's ~3.6 GB, raising the roofline ceiling from 6.67 to 10.9 tok/s.

3. **Further reduces effective bytes via FATReLU**: 85% FFN sparsity skips the majority of down_proj computation, the largest single GEMV per layer.

4. **Maintains quality**: Salience allocation preserves precision where it matters most (attention projections, early/late layers), keeping PPL within +1.0 of uniform Q4.

5. **No architectural dead ends**: Each step is independently testable and provides monotonic improvement. If any step fails its gate, we diagnose and fix before accumulating technical debt.
